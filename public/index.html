2.3 Justificación Arquitectónica del Diseño

La segmentación física y lógica en 4 redes Docker separadas responde de manera rigurosa al principio de mínimo privilegio y a las mejores prácticas de defensa en profundidad de la ciberseguridad corporativa:

Por ejemplo, el contenedor expuesto directamente a Internet (men-web) está completamente aislado y no posee rutas ni capacidad de direccionamiento directo hacia la subred de base de datos (red-db). Las transacciones y consultas son controladas, previniendo intrusiones directas en caso de un ataque de inyección web. Adicionalmente, todos los flujos de administración técnica e intercambio remoto de ficheros NFS operan de forma encapsulada y aislada.

Esto refleja fielmente las normas de resiliencia y resguardo de datos exigidas por el Ministerio de Educación Nacional (MEN), donde el resguardo, confidencialidad e integridad de la información académica y registros de millones de ciudadanos colombianos es una prioridad crítica e ineludible.

Justificación Técnica del RAID 1:

El mirroring de discos (RAID 1) fue elegido porque el Ministerio de Educación Nacional (MEN) custodia registros académicos y datos personales de millones de ciudadanos colombianos, donde la disponibilidad absoluta y la integridad total de la información son prioridades inquebrantables por encima de la velocidad de escritura pura. La redundancia en espejo garantiza recuperación inmediata y automática ante fallas de disco rígido sin pérdida alguna de datos ni interrupción de servicios.

📄 6.1 Script de Backup (backup.sh)
Rotación 7 días
#!/bin/bash
FECHA=$(date +%Y%m%d_%H%M%S)
DEST=/backups && LOG=/logs/backup.log

echo "[$FECHA] Iniciando backup..." >> $LOG
docker exec men-db mysqldump -u root -pmen_root_2024 men_db > $DEST/db_$FECHA.sql
tar -czf $DEST/configs_$FECHA.tar.gz /home/ubuntu/proyecto-men/services/
find $DEST -name '*.sql' -mtime +7 -delete
find $DEST -name '*.tar.gz' -mtime +7 -delete
echo "[$FECHA] Backup completado" >> $LOG
📄 6.2 Script de Monitoreo (monitor.sh)
Alerta Automática
#!/bin/bash
LOG=/logs/monitor.log && FECHA=$(date +%Y%m%d_%H%M%S)

docker ps --format '{{.Names}}: {{.Status}}' >> $LOG
CPU=$(top -bn1 | grep 'Cpu(s)' | awk '{print $2}')
RAM=$(free -m | awk 'NR==2{printf "%.0f", $3*100/$2}')
DISCO=$(df -h / | awk 'NR==2{print $5}' | cut -d'%' -f1)
echo "[$FECHA] CPU: $CPU% | RAM: $RAM% | Disco: $DISCO%" >> $LOG
if [ "$DISCO" -gt 90 ]; then
    echo "[$FECHA] ALERTA: Disco al $DISCO%" >> $LOG
fi

# Configuración de balanceo en /etc/nginx/conf.d/men.conf
upstream men_backend {
    server 172.20.1.10:80 weight=1;         # Servidor primario
    server 172.20.1.11:80 weight=1 backup;  # Failover automático
}
server {
    listen 80;
    location / {
        proxy_pass http://men_backend;
    }
}

Si el servidor primario (172.20.1.10) falla, Nginx redirige automáticamente el tráfico al servidor de respaldo (172.20.1.11) sin interrupción del servicio. Esta configuración implementa failover automático apropiado para los servicios del MEN.

Nota Técnica (docker kill vs kill -9):

El comando docker kill envía una señal explícita a la API de Docker para detener el contenedor; al ser intencional, la política restart: always NO reinicia el servicio. Sin embargo, utilizar kill -9 <PID> directamente sobre el proceso en el host (Linux) simula una caída abrupta o un "crash" inesperado del sistema. En este escenario, el daemon de Docker detecta que el proceso murió sin intervención de la API e invoca exitosamente la política de reinicio.

Anexo Técnico

Documentación y Decisiones

Registro arquitectónico de las decisiones técnicas adoptadas a lo largo del proyecto, evaluando alternativas y justificando el diseño implementado.

Fase 1 · Entorno

Sistema Operativo y Virtualización

▼
¿Por qué Ubuntu 26.04 LTS y no otra distro?
Ubuntu LTS (Long Term Support) ofrece 5 años de actualizaciones de seguridad garantizadas, estabilidad empresarial y excelente soporte nativo para contenedores Docker. Es el estándar de facto en la industria Cloud moderna.
Alternativas descartadas: Debian (ciclos de actualización más lentos), CentOS (descontinuado como LTS estable).
💡 "Priorizamos estabilidad y ciclo de vida extendido para evitar refactorizaciones forzadas a corto plazo."
¿Por qué VirtualBox con adaptador NAT + Host-Only?
El adaptador NAT permite la salida segura de la VM a Internet para descargar paquetes, mientras que Host-Only establece una red privada, aislada e invisible desde el exterior, exclusiva para la administración SSH desde la máquina anfitriona.
Alternativas descartadas: Modo Bridge (expondría la VM directamente a la red local comprometiendo la seguridad).
Fase 2 · Red

Arquitectura y Segmentación

▼
¿Por qué 4 subredes separadas y no una sola plana?
La segmentación previene el movimiento lateral en caso de vulneración. Si un atacante compromete el servidor web, no podrá escanear ni acceder directamente al puerto 3306 de la base de datos o al puerto 2222 de administración.
Alternativas descartadas: Red plana / default bridge de Docker (no aplica principio de mínimo privilegio).
¿Por qué utilizar el rango 172.20.x.x?
Es parte del espacio de direcciones IP privadas (Clase B - RFC 1918) asignado típicamente para redes de contenedores empresariales. Elegimos la subred 20 para evitar conflictos con el rango 172.17.x.x por defecto de Docker.
🔒 "Aplicación estricta del Principio de Mínimo Privilegio (PoLP) en la topología de red."
Fase 3 · Docker

Orquestación de Microservicios

▼
¿Por qué Docker Compose y no Kubernetes?
Docker Compose es la herramienta idónea para implementaciones de un solo host (single-node). Kubernetes agregaría una sobrecarga innecesaria de recursos (control plane, etcd) e incrementaría drásticamente la complejidad operativa para una VM aislada.
Alternativas: Kubernetes, Docker Swarm (mejores para multi-node).
Justificación de las imágenes seleccionadas
nginx: Altamente eficiente para proxy inverso y balanceo estático.
mysql:8.0: Estabilidad comprobada y motor relacional ACID necesario para datos del MEN.
linuxserver/openssh: Imagen aligerada con soporte Alpine, ideal para un servidor bastión.
cturra/ntp & erichough/nfs: Imágenes minimalistas, sin daemon systemd, específicas para su único microservicio.
Política 'restart: always' vs 'restart: unless-stopped'
Configuramos restart: always para garantizar que si el daemon de Docker se reinicia (por actualización o reboot de la VM), los contenedores del MEN vuelvan a levantarse sin necesidad de ejecutar docker compose up manualmente.
¿Por qué men-web2 necesita un nginx.conf propio mapeado?
A diferencia del Nginx principal, el servidor de backup no debe contener las directivas del upstream (para evitar un bucle de proxy). Al mapear su propio default.conf vacío, aseguramos que solo entregue contenido local y no intente balancear carga de nuevo.
Fase 4 · RAID + LVM

Almacenamiento Redundante

▼
¿Por qué RAID 1 y no RAID 0 o RAID 5?
RAID 1 provee espejeo exacto (Mirroring). Si falla un disco, el otro contiene toda la data, ideal para el Ministerio de Educación donde la pérdida de registros es inaceptable. RAID 0 no tiene tolerancia a fallos, y RAID 5 requiere mínimo 3 discos y gasta más CPU calculando paridad.
Alternativas descartadas: RAID 0 (prioriza velocidad pero sin respaldo), RAID 5 (overhead de paridad).
¿Por qué LVM sobre el arreglo en lugar de particiones fijas?
LVM (Logical Volume Manager) nos brinda flexibilidad. Si en 6 meses la base de datos de instituciones colapsa el espacio, podemos añadir otro disco al RAID y extender el volumen lv-data en caliente, sin apagar el servidor.
Distribución de tamaños: /data (3GB), /backups (3GB), /logs (2GB)
Asignamos cuotas controladas para evitar que, si un servicio se descontrola (ej. logs masivos), consuma todo el disco y afecte a la base de datos. Dejamos espacio libre en el Volume Group para expansiones futuras.
📈 "LVM nos asegura escalabilidad transparente de almacenamiento a largo plazo."
Fase 5 · Seguridad

Bastionado del Servidor

▼
¿Por qué usar UFW en lugar de configurar iptables directamente?
UFW (Uncomplicated Firewall) es un frontend para iptables que reduce el riesgo de errores humanos al crear reglas de red. Facilita la auditoría rápida y la gestión de la política 'Default Deny' para todo el tráfico entrante no explícito.
Alternativas descartadas: Reglas puras de iptables (mayor curva de aprendizaje y propensión a bloquear accesos críticos por error).
¿Por qué SSH con llave pública RSA 4096 bits en vez de contraseña?
Las contraseñas son vulnerables a ataques de diccionario y fuerza bruta automatizada. Una llave criptográfica RSA de 4096 bits hace matemáticamente inviable descifrar el acceso en un tiempo útil, y mitiga la interceptación de credenciales (Man-in-the-Middle).
División de roles de usuarios
Se implementó separación de perfiles para cumplir la regla de oro: cada operador o tarea automática debe poseer únicamente los privilegios estrictamente necesarios para su función, minimizando el impacto si una cuenta es comprometida.
Fase 6 · Scripts

Automatización de Operaciones

▼
¿Por qué usar scripts en Bash y no en Python?
Bash es nativo, no requiere instalación de intérpretes adicionales ni dependencias. Para tareas del sistema interactuando directamente con ejecutables como docker exec o tar, Bash es la herramienta más eficiente y directa.
Política de rotación de 7 días
Se elimina silenciosamente cualquier respaldo o log que exceda los 7 días (mediante el comando find -mtime +7). Esto asegura que el particionamiento de los LVM (como el lv-backups) nunca se sature de forma descontrolada.
Programación de tareas (Crontab)
Se estableció el backup pesado a las 02:00 AM, horario de menor concurrencia de estudiantes y docentes en el sitio. Por el contrario, el monitor de recursos corre cada 5 minutos de forma perpetua para generar alertas en tiempo real.
Fase 7 · Alta Disponibilidad

Failover y Resiliencia

▼
¿Por qué un upstream con directiva 'backup' y no Round-Robin puro?
En Round-Robin, Nginx divide el tráfico 50/50. Sin embargo, al configurar el nodo 2 con el tag backup, garantizamos que el 100% de los usuarios vayan al nodo principal con mayor capacidad computacional, mientras que el secundario permanece en estado inactivo sin consumir recursos, despertando ÚNICAMENTE si el primario cae.
Alternativas: Round-robin (distribuye carga equitativamente), IP-hash (fija sesiones).
Diferencia entre 'docker kill' y 'kill -9 PID' para pruebas HA
docker kill es una orden administrativa enviada a la API de Docker; al ser intencional, NO invoca la política restart: always. Para verificar realmente la resiliencia ante un "Kernel Panic" o falla abrupta, se debe utilizar la señal SIGKILL (kill -9) directo al PID de Linux en el host.
⚠️ "Validamos fallas simulando condiciones reales de caída de procesos a nivel de Kernel."
